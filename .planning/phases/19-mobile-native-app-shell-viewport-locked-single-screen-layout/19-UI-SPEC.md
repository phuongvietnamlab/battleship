# Phase 19 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope reminder:** This is a LAYOUT-STRUCTURE redesign of an EXISTING app, not a visual redesign. The dark-ocean brand, gold accents, glass panels, fonts, animations, and CSS custom properties in `public/style.css` are LOCKED AS-IS. This contract governs only the new app-shell regions (header / main / footer-actionbar), viewport-fit rules, overlay/sheet patterns for overflow content, and the small set of new controls (back buttons, overlay toggles) needed to satisfy MOBILE-01..12.

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (hand-rolled CSS, no Tailwind/shadcn) |
| Icon library | emoji glyphs (existing convention: ⚓ 💬 📋 👥 ✕ ← 🔊/🔇) |

**Rationale for `Tool: none`:** CLAUDE.md constraint: "Stay on Node.js + Express + Socket.IO + React + esbuild — extend, don't rewrite." The project has no Tailwind config, no `components.json`, and uses a single hand-authored `public/style.css` (1476 lines) with CSS custom properties as its design-token system. Introducing shadcn would require a build-pipeline change (Tailwind/PostCSS) that contradicts "extend, don't rewrite" and is out of scope for a pure layout-restructure phase. All tokens below are extracted from the existing `:root` block and established class patterns.

## Spacing Scale

Declared values (must be multiples of 4) — matches existing usage in `style.css`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, tight inline spacing |
| sm | 8px | Compact element spacing, gap between header controls |
| md | 14px | **App shell side padding** (existing `.app { padding: 14px 14px 60px }` — keep 14px, not 16px, to avoid relayout of every existing screen) |
| lg | 16px | Default element spacing, card padding |
| xl | 24px | Section padding (e.g. `.lobby`/`.profile-view` padding) |
| 2xl | 32px | Major section breaks (modal padding) |

Exceptions:
- **14px side padding** (not the canonical 16px) — existing `.app` padding is 14px; changing it would shift every screen's horizontal rhythm and is out of scope for a layout-shell phase. New shell regions (header bar, footer action bar) MUST reuse this 14px gutter for visual continuity.
- **44px minimum touch target** — all new tap targets introduced by this phase (back buttons, overlay-open chips, tab triggers) MUST be ≥44×44px hit area even if the visible icon is smaller (use padding to pad out the hit area). Existing `.btn` (padding:15px) and `.btn.ghost` (padding:5-8px on mobile) already vary; new controls follow the 44px rule via `min-height`/`min-width` regardless of visual padding.
- **Safe-area insets** — `env(safe-area-inset-top/right/bottom/left)` MUST be added to the shell header (top) and footer/action-bar (bottom) per MOBILE-07. This is additive to the spacing scale, not a replacement.

> **Planner note:** The 14px gutter is the ONLY justified non-canonical value. No NEW non-standard spacing values may be introduced in shell CSS beyond this 14px reuse and the 44px touch-target minimum. All other spacing uses the 4/8/16/24/32 scale above.

## App Shell Layout Contract (MOBILE-01 / 02 / 03 / 08)

**Visual hierarchy intent:** Across all shell regions, the `.shell-main` content area is always the primary visual focus (highest contrast, the ocean background, the board/lobby/list). The `.shell-header` and `.shell-footer` are chrome — lower-contrast `--panel` glass that recedes — present for orientation and navigation, never competing with main content for attention. Accent `--gold` appears in chrome only to mark the single active/primary action.

### Root lock
- `html, body`: `height: 100dvh`, `overflow: hidden` (replaces current `min-height: 100dvh` which permits page scroll). `overscroll-behavior: none` (already present) stays.
- `#root` and the existing `.app` div become the shell root: `display: flex; flex-direction: column; height: 100dvh; height: 100vh /* fallback */;` — `.app` keeps `max-width: 480px; margin: 0 auto;` (desktop phone-frame, MOBILE-08) but padding changes from `14px 14px 60px` to a 3-region split below.

### Three shell regions (MOBILE-02)
Every screen renders inside this structure (a new wrapper component, e.g. `<ScreenShell header={...} footer={...}>{children}</ScreenShell>`):

1. **`.shell-header`** — fixed height, does not scroll.
   - Padding: `8px 14px`, plus `padding-top: calc(8px + env(safe-area-inset-top, 0px))`.
   - Background: `--panel` + existing blur (reuses `.lobby/.room-banner/.modal/.log` glass rule).
   - Content: screen title (22px/700/Oswald) left-aligned or centered, optional back button (left, ← arrow, `.btn.ghost`, 44px hit area), optional right-side action slot (sound toggle, avatar chip — reuses existing `.topbar-right` contents).
   - On the **lobby** screen specifically, the header is the existing `.topbar` (logo badge + tagline + avatar/sound) — unchanged content, just now pinned as the shell header region instead of flowing into scrollable content.

2. **`.shell-main`** — `flex: 1; min-height: 0; overflow-y: auto; overflow-x: hidden;` padding `14px`.
   - This is the ONLY region that may scroll, and only when its content genuinely exceeds available height (e.g. a long match-history list, a tall help modal body).
   - For screens designed to fit without scroll (lobby, room, queue, profile, friends-empty), `overflow-y: auto` is a safety net, not the primary fit strategy — primary fit comes from sizing content to `100%` of available height via flex.
   - **Battle screen** (MOBILE-03): `.shell-main` becomes a flex column; the `.boards`/board area is sized `min(100% - other-elements-height, ...)` so board + scoreboard + turn ring fit without scroll. The existing `--cell: min(32px, calc((100vw - 40px) / 11))` formula is extended to also cap by available height: `--cell: min(32px, calc((100vw - 40px) / 11), calc((var(--main-h, 100vh) - 220px) / 11))` where `--main-h` is set via JS from `.shell-main`'s measured height (or `ResizeObserver`).

3. **`.shell-footer`** — fixed height, does not scroll, optional (omit entirely if a screen has no persistent footer actions).
   - Padding: `10px 14px`, plus `padding-bottom: calc(10px + env(safe-area-inset-bottom, 0px))`.
   - Background: `--panel` + blur, `border-top: 1px solid var(--panel-brd)`.
   - Content: primary action button(s) (e.g. lobby's hero CTA stays in `.shell-main` per LOBBY-07 — it is NOT moved to the footer, since Phase 9 already fit it on one viewport; the footer is reserved for screens that need a persistent bottom action, e.g. Placement's "Confirm" button, Battle's "End Turn"/power-up overlay toggle, Friends' "Add friend" FAB-equivalent).
   - The existing `.footer-note` (copyright/seo link) is REMOVED from the always-rendered shell on mobile (it pushed content below the fold) and relocated to a "Settings/About" entry inside the avatar menu or a dedicated overlay — not part of the primary shell.

### Per-screen shell mapping

| Screen | Header content | Main content | Footer content |
|--------|----------------|---------------|-----------------|
| lobby | Existing `.topbar` (logo + avatar/balance + sound) | Hero CTA + 2 lobby cards + mode toggle + friends widget (unchanged from Phase 9, just height-constrained to `.shell-main`) | none |
| room | Title "Room" + back button (→ leaveRoom confirm) | Room code box, invite link, waiting pill, start-placement button | none |
| placement | Title (room code / vs-bot label) + status pill (existing `.room-banner` content, compacted) | Placement board + hint + power-up shop row | "Confirm placement" primary button (existing confirm button moves here) |
| battle | Scoreboard + turn ring (existing `.scoreboard`, compacted to header height) | Boards (sized to fit, MOBILE-03) | Tap-open overlay toggles: Power-ups, Battle Log, Chat (see Overlay contract below) |
| profile | Title "Profile" + back button | Avatar/name/stats (existing `.profile-view` content) | Action buttons row (existing `.profile-actions`) |
| history | Title "Match History" + back button (existing `.history-header`) | Scrollable match-card list (genuinely scrolls — list can exceed viewport) | none |
| friends | Title "Friends" + back button (existing `.friends-header`) | Search + friend list (may scroll if >~6 friends) | none |
| queue | Title "Finding match" (no back; cancel is the action) | Elapsed timer + searching pill | "Cancel" button |

## Overlay / Bottom-Sheet Contract (MOBILE-04)

Battle-screen content that previously stacked below the boards and pushed the page past the fold becomes tap-open overlays, reusing the EXISTING `.bottom-sheet-overlay` / `.bottom-sheet-panel` pattern (already defined in `style.css:260-277`, used by Phase 9's lobby bottom sheets). No new overlay CSS pattern is introduced — only new trigger chips and content mappings.

| Content | Old location | New trigger (footer chip) | Overlay content |
|---------|-------------|----------------------------|-------------------|
| Power-up bar (`.power-bar`/`PowerBar`) | Stacked above enemy grid, always visible | Footer chip: icon `⚡` + label "Powers" / "Vũ khí" + badge showing count of usable power-ups (gold badge, reuses `.power-btn .pc` badge style) | Bottom sheet listing `PowerBar` buttons full-width (one per row, `.sheet-option` style) |
| Battle log (`.log`) | Block below boards, `max-height:130px overflow:auto` | Footer chip: icon `📜` + label "Log" / "Nhật ký"; shows a gold dot badge when a new log entry arrives while sheet is closed | Bottom sheet containing the existing `.log` list (full `.bottom-sheet-panel`, scrollable to 70dvh) |
| Chat composer | Already a fixed bottom panel (`.chat-panel`) | Existing `.chat-toggle` button in `.roombar`/header — UNCHANGED, already overlay-based | unchanged |

Footer chip styling: reuse `.btn.ghost.compact` (existing `.btn.compact { padding:10px 14px; font-size:13px }` + `.btn.ghost` colors). Active/open state adds `border-color: var(--gold); color: var(--gold)`. Badge dot/count reuses `.power-btn .pc` (gold pill, `position:absolute`).

Opening an overlay does NOT navigate (`screen` state unchanged) — it is local component state (`useState`) toggling the bottom-sheet `open` prop, identical to how `BottomSheet` is already used in Phase 9's lobby.

## Navigation & Transitions (MOBILE-06)

- Screen changes (`setScreen(...)`) get a slide transition: outgoing screen slides out `translateX(-20%)` + fade, incoming slides in from `translateX(20%)` + fade, duration 220ms, `cubic-bezier(.34,1.4,.4,1)` (reuses the existing `rise` easing curve family already in `style.css`).
- Back navigation (profile/history/friends → lobby) slides the reverse direction.
- Wrap the transition in `@media (prefers-reduced-motion: no-preference)` — under `reduce`, screens swap with an opacity-only 100ms cross-fade (matches the existing reduced-motion pattern at `style.css:489-493`).
- Implementation note for planner/executor: a single CSS class toggle (`.screen-enter` / `.screen-exit`) applied to the shell-main wrapper, driven by a `direction` ref tracked alongside `setScreen`, is sufficient — no router/animation library needed (consistent with "extend, don't rewrite").

## Keyboard / Safe-Area Handling (MOBILE-07 / MOBILE-12)

- Shell header: `padding-top: calc(8px + env(safe-area-inset-top, 0px))`.
- Shell footer: `padding-bottom: calc(10px + env(safe-area-inset-bottom, 0px))`.
- Left/right safe-area insets applied to `.app` itself: `padding-left: max(14px, env(safe-area-inset-left)); padding-right: max(14px, env(safe-area-inset-right));` (covers landscape notch cases).
- Chat composer (`.chat-panel`, already `position:fixed; bottom:0`): when the on-screen keyboard opens, rely on `100dvh` recalculation (existing `max-height:70dvh` already adapts); additionally listen to `window.visualViewport.resize` to re-clamp `.chat-panel` `bottom` offset to `0` so it never sits beneath the keyboard on iOS Safari (known iOS quirk where `position:fixed` + keyboard misplaces fixed elements). This is a JS behavior note for the executor, not a new visual token.

## Responsive / No-Horizontal-Scroll (MOBILE-11)

- All shell regions: `max-width: 100%; overflow-x: hidden;`.
- Long names (player names, room codes, friend names) continue to use existing truncation rules (`.pc-name`, `.friend-name` already have `overflow:hidden; text-overflow:ellipsis; white-space:nowrap`) — extend the same `text-overflow: ellipsis` treatment to the new shell-header title when a screen title + back button + right-action would otherwise overflow narrow widths (e.g. 320px).
- Verify at 360×640, 390×844, 414×896 per MOBILE-05 — no new breakpoints beyond the existing `@media (max-width:560px)`, `@media (max-width:380px)`, `@media (max-width:320px)` already in `style.css`.

## Open Questions (flagged, not blocking — defaults chosen below)

1. **Lobby persistent bottom tab-bar?** DEFAULT: NO. The lobby's hero CTA + 2 cards already fit one viewport per Phase 9 (LOBBY-07). Adding a persistent tab bar would consume ~56px of vertical budget on the smallest target (360×640) and risks breaking that fit. Profile/History/Friends remain accessible via the existing avatar menu (Phase 9 LOBBY-10), now also reachable via back-button navigation rather than a tab bar. If a future phase wants a tab bar, it should be scoped explicitly with a re-verified LOBBY-07 budget.
2. **Battle-screen footer chips when `vsBot`** — DEFAULT: show "Powers" chip only if `inv` has any non-zero count (matches existing `PowerBar` conditional rendering); "Log" chip always shown. Chat toggle hidden for `vsBot` (matches existing `{!vsBot && <ChatComposer .../>}` guard).
3. **Desktop (>560px) behavior for overlays** — DEFAULT: bottom sheets remain bottom-anchored even on desktop (consistent with existing Phase 9 `BottomSheet` component, which is already width-capped via `.app`'s 480px frame), no separate desktop popover variant introduced.
