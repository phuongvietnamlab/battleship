## Locked decision: leaveRoom uses INLINE recording, NOT endGameForfeit routing

Research Open Question 1 / Assumption A2 flagged that routing `leaveRoom` through `endGameForfeit` would change client-visible behavior: the surviving player would receive `gameOver({win:true})` instead of today's `opponentLeft`. Verified in public/app.jsx (lines 1727-1733): `gameOver` and `opponentLeft` are handled by SEPARATE listeners with different UX (gameOver shows the win/lose overlay; opponentLeft shows a log + flag and does NOT open the overlay). To preserve existing behavior, **leaveRoom records the match INLINE** (reads room.started + derives winnerId + calls recordMatch before mutating the seat) and KEEPS its existing `opponentLeft` emit. Do NOT route leaveRoom through endGameForfeit.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client socket event (fire/leaveRoom/disconnect) → server end path | the trigger is client-initiated but the outcome (winner/loser/reason) is derived 100% from server room state; no client-supplied result is trusted |
| server end path → recordMatch | passes only server-derived integer userIds + a fixed-taxonomy reason |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-06 | Spoofing / Tampering | client forging a match outcome by emitting a crafted event | HIGH | mitigate | recordMatch is called ONLY from server-authoritative end paths; winner/loser/reason are derived from room state and opponentOf(), never read from the socket payload. There is no client event that can directly trigger a match write. |
| T-03-07 | Tampering | duplicate / double-write under racing end paths (e.g. win + disconnect same tick) | MEDIUM | mitigate | `room.recorded = true` set synchronously before the recordMatch promise (single-threaded event loop guarantees first-writer-wins, D-06); DB UNIQUE (Plan 01) is the backstop |
| T-03-08 | Repudiation | grace-expiry forfeit not attributed to the right loser | MEDIUM | mitigate | userId captured on the seat at join (Task 1) and read BEFORE seat deletion (Pitfall 2/6); loser = the absent clientId, winner = opponentOf |
| T-03-09 | Denial of Service | slow/failed DB write freezing the end-game screen | MEDIUM | mitigate | gameOver emitted first; recordMatch is fire-and-forget `.catch(()=>{})`; recordMatch itself no-ops/swallows (Plan 02) — D-07 |
| T-03-SC | Tampering | npm installs | N/A | accept | No new packages this phase (03-RESEARCH Package Legitimacy Audit — none) |
