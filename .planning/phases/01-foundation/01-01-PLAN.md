<objective>
Stand up the durable persistence slice: a single shared Postgres pool (`db.js`), an auto-applied numbered-migration runner that lays down the canonical identity schema (`users` + `credentials` + `schema_migrations`), and a transparent guest-credential upsert wired into the existing `clientId` connect flow. Also scaffold the project's first test harness (Vitest) so every later task in this phase has an `<automated>` verify target.

This is a complete vertical persistence slice: after it lands, a real guest connecting to the server produces a durable DB row, the schema self-provisions on deploy, and the server fails loud on a bad migration. Delivers DATA-01 and DATA-02.

Purpose: Postgres persistence is the root dependency that unlocks accounts, match recording, and ranked play in later phases. The identity model (one `users`, many `credentials`) is laid down now per PITFALLS Pitfall 1 to avoid a painful retrofit when Google OAuth lands in Phase 2.
Output: `vitest.config.js`, `test/` scaffold, `db.js`, `migrations/001_identity.sql`, `server.js` boot + connect-path modifications, `pg` + `vitest` added to package.json.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser clientId → server upsert | Untrusted localStorage clientId crosses into a DB write |
| env → Postgres connection | Connection params (secrets) come from env; SSL posture is env-gated |
| server → Postgres | All SQL crosses the process boundary into the DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Tampering/Injection | upsertGuestCredential SQL path | mitigate | Parameterized query only — clientId bound as `$1`, never string-concatenated. Migration files are static DDL with no user input. |
| T-01-D1 | Denial of Service | pg connection pool | mitigate | Single shared pool (`max: 10`) per D-05 — prevents per-request pool creation / connection exhaustion (PITFALLS Pitfall 4). |
| T-01-I1 | Information Disclosure | DB credentials | accept | Secrets sourced from env only; not logged. Localhost EC2 connection, SSL env-gated for future remote use. |
| T-01-A1 | Availability | guest DB write | mitigate | upsertGuestCredential is fire-and-forget with caught errors — a DB outage degrades to RAM-only guest play, never crashes the game loop. |
| T-01-T2 | Tampering | half-migrated schema | mitigate | Fail-loud runner (D-02): a bad migration aborts boot before listen(), so the server never serves on an inconsistent schema. |
| T-01-SC | Tampering | npm installs (`pg`, `vitest`) | mitigate | Task 0 blocking-human checkpoint verifies both packages on npmjs.com before install (no RESEARCH audit table exists). |
</threat_model>
