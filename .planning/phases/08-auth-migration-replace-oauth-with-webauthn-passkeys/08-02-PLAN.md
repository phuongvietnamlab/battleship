       ```js
       // ─── WebAuthn credential functions ────────────────────────────────────────────

       async function getWebAuthnCredentials(userId) {
         const { rows } = await pool.query(
           "SELECT id, public_key, counter, transports FROM webauthn_credentials WHERE user_id=$1",
           [userId]
         );
         return rows;
       }

       async function getWebAuthnCredentialById(credentialId) {
         const { rows } = await pool.query(
           "SELECT id, user_id, public_key, counter, transports FROM webauthn_credentials WHERE id=$1",
           [credentialId]
         );
         return rows[0] || null;
       }

       async function saveWebAuthnCredential({ id, userId, publicKey, counter, transports, deviceName }) {
         await pool.query(
           `INSERT INTO webauthn_credentials (id, user_id, public_key, counter, transports, device_name)
            VALUES ($1, $2, $3, $4, $5, $6)`,
           [id, userId, publicKey, counter, transports || [], deviceName || null]
         );
       }

       async function updateWebAuthnCounter(credentialId, newCounter) {
         await pool.query(
           "UPDATE webauthn_credentials SET counter=$1 WHERE id=$2",
           [newCounter, credentialId]
         );
       }
       ```

    ```js
    // ─── WebAuthn (Passkey) routes ───────────────────────────────────────────────
    const { generateRegistrationOptions, verifyRegistrationResponse,
            generateAuthenticationOptions, verifyAuthenticationResponse } = require("@simplewebauthn/server");
    const { getWebAuthnCredentials, getWebAuthnCredentialById,
            saveWebAuthnCredential, updateWebAuthnCounter } = require("./db");

    // RP config from env (D-06)
    const RP_NAME = process.env.RP_NAME || "Battleship Online";
    const RP_ID = process.env.RP_ID || (process.env.SITE_ORIGIN ? new URL(process.env.SITE_ORIGIN).hostname : "localhost");
    const RP_ORIGIN = process.env.RP_ORIGIN || process.env.SITE_ORIGIN || "http://localhost:4000";
    ```

    **Route 1: POST /auth/webauthn/register-options**
    - Requires authenticated user (req.user) OR accepts a guest clientId for new account creation
    - If req.user: generate registration options excluding existing credentials
    - If no req.user but clientId provided: create a new user, link guest (reuse linkOrPromoteAccount pattern), then generate options
    - Store challenge in req.session.webauthnChallenge
    - Return options JSON

    **Route 2: POST /auth/webauthn/register-verify**
    - Read challenge from req.session.webauthnChallenge
    - Call verifyRegistrationResponse with expected challenge, origin (RP_ORIGIN), rpID (RP_ID)
    - On success: save credential via saveWebAuthnCredential
    - Clear req.session.webauthnChallenge
    - Return { ok: true, credentialId }

    **Route 3: POST /auth/webauthn/login-options**
    - No auth required (this is for signing in)
    - Generate authentication options (allowCredentials empty = discoverable credential / resident key)
    - Store challenge in session
    - Return options JSON

    **Route 4: POST /auth/webauthn/login-verify**
    - Read challenge from session
    - Extract credential ID from response body
    - Look up credential in DB via getWebAuthnCredentialById
    - Call verifyAuthenticationResponse with stored public key, counter, expected challenge/origin/rpID
    - On success: update counter, establish session (regenerate → stamp user_id → save)
    - Return { ok: true, user: { id, displayName, avatarUrl } }

    All routes use authRateLimit. All return JSON. Error responses use structured codes: WEBAUTHN_FAILED, NOT_AUTHENTICATED, CREDENTIAL_NOT_FOUND.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server (registration) | Client sends attestation response; server verifies against expected challenge/origin/RP |
| browser → server (authentication) | Client sends assertion; server verifies signature, counter, and credential ownership |
| session challenge storage | Challenge stored server-side only; client cannot forge or replay |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | Spoofing | WebAuthn login | mitigate | Server verifies assertion signature against stored public key; credential must exist in DB for the claimed user. RP_ORIGIN and RP_ID validation prevent cross-site credential theft. |
| T-08-05 | Replay | authentication assertion | mitigate | Signature counter incremented and stored; replayed assertions with old counter are rejected. Challenge is single-use (cleared from session after verify). |
| T-08-06 | Tampering | registration response | mitigate | verifyRegistrationResponse checks origin + rpID match; attestation format validated by @simplewebauthn/server. Challenge verified against session-stored value. |
| T-08-07 | Elevation | guest → account via passkey | mitigate | New account creation through passkey follows same linkOrPromoteAccount flow as email signup — no privilege bypass. Rate-limited. |
| T-08-08 | DoS | WebAuthn endpoints | mitigate | authRateLimit (10/min per IP) applied to all four routes. Challenge generation is cheap (random bytes). |
| T-08-09 | Information Disclosure | credential enumeration | mitigate | login-options returns empty allowCredentials (discoverable credentials mode) — does not reveal which users have passkeys. |
</threat_model>
