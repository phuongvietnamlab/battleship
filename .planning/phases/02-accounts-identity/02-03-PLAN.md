<objective>
The sessions + revocation vertical slice: the signed-in header UI (avatar chip + dropdown) plus server-side sign-out and sign-out-all-devices. This completes AUTH-04 — a signed-in player stays logged in across visits (the 30-day rolling cookie already delivered by Plan 02) and can revoke access from this device or everywhere.

Builds directly on Plan 02's authUser state and session plumbing: adds the avatar chip rendering (replacing the sign-in button when signed in), the dropdown menu with the inline "sign out everywhere?" confirmation, and the two revocation routes. The sign-out-all path uses the indexed session.user_id column created in Plan 01's migration, stamped at login inside Plan 02's named callback success handler (onGoogleCallbackSuccess).

Purpose: Server-side revocation is the AUTH-04 acceptance gate and a security requirement (a leaked/old session must be killable). The single-DELETE-by-user_id approach (D-03) is atomic and race-free — but it only works if every login row carries user_id, which this plan stamps.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /auth/signout-all | revocation request; must be tied to the authenticated req.user, never a client-supplied user_id |
| session row -> user_id column | the DELETE key; written server-side at login only (stamped in onGoogleCallbackSuccess) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-12 | Elevation of Privilege | sign-out-all targeting another user | mitigate | userId taken from req.user.id only (401 if absent); never from request body — client cannot revoke others |
| T-02-13 | Elevation of Privilege | stale session after sign-out-all (race) | mitigate | single indexed DELETE FROM session WHERE user_id=$1 removes all rows atomically; subsequent requests find no session |
| T-02-14 | Tampering | sign-out-all SQL injection via user_id | mitigate | parameterized $1 binding; user_id is an integer from req.user |
| T-02-15 | Spoofing | sign-out fails silently (sync logout) | mitigate | req.logout(callback) — Passport 0.6+ async; session.destroy only after logout resolves (Pitfall 3) |
| T-02-20 | Elevation of Privilege | sign-out-all finds zero rows (unstamped session) | mitigate | login stamps req.session.user_id = req.user.id before req.session.save inside onGoogleCallbackSuccess, so every persisted session row carries the DELETE key |
</threat_model>
