<objective>
The password-reset slice (AUTH-08): request a reset, receive a single-use time-limited emailed token, set a new password. It reuses every primitive already built — the Plan 06 auth_tokens storage (purpose='reset', 1h ttl) + createAuthToken/consumeAuthToken, the Plan 08 graceful-degrading mailer, and the Plan 06/07 bcrypt hashing — adding only a new password-setter db helper, two routes, and the reset UI behind the "Forgot password?" link already placed in Plan 07.

The two security-critical properties: the request endpoint must be enumeration-safe (identical response whether the email exists or not), and the reset token must be single-use, time-limited, and tamper-evident (already guaranteed by Plan 06's consumeAuthToken). Setting the new password reuses the same bcrypt cost + min-8 enforcement as signup.

Purpose: AUTH-08 completes the email-account lifecycle (recover a forgotten password) using the standard tokenized-email flow (D-19) with no new token machinery — proving the Plan 06 auth_tokens design serves both verification and reset.
Output: db.js (setEmailPassword), server.js (reset-request + reset routes), public/app.jsx (PasswordResetForm + EN/VI), test/auth.test.js (AUTH-08 assertions).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /auth/reset-request | untrusted email; response must not reveal account existence |
| emailed link -> ?reset=token -> POST /auth/reset | token crosses back as untrusted input; must be single-use, expiring, tamper-evident |
| reset success -> existing sessions | a successful reset should not leave a hijacked old session valid |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-44 | Information Disclosure | account enumeration via reset-request | mitigate | /auth/reset-request returns an identical generic ok:true (and the UI shows the same message) whether or not the email exists; email send is conditional + silent (D-19) |
| T-02-45 | Tampering | reset-token forgery / replay | mitigate | consumeAuthToken('reset') enforces single-use (conditional consumed_at UPDATE) + 1h expiry + 256-bit entropy (Plan 06); forged/expired/reused -> BAD_TOKEN |
| T-02-46 | Tampering | weak new password via reset | mitigate | setEmailPassword enforces min 8 -> WEAK_PASSWORD server-side (same as signup) |
| T-02-47 | Elevation of Privilege | old/leaked session survives a password reset | mitigate | on successful reset, DELETE FROM session WHERE user_id=$1 invalidates all existing sessions for that user (reuses Plan 03 pattern) |
| T-02-48 | Denial of Service | reset-request / reset brute force | mitigate | authRateLimit (RateLimiterMemory 10/60s per IP) on both routes |
| T-02-49 | Tampering | SQL injection via email/token/password | mitigate | all DB access via parameterized helpers (createAuthToken/consumeAuthToken/setEmailPassword); routes never build SQL |
</threat_model>
