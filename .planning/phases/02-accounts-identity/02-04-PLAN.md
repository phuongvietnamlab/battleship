<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> GET /api/profile/:id | public read endpoint; :id is untrusted input crossing into a SQL query |
| users row -> profile JSON | server controls which columns are exposed publicly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | Tampering | /api/profile/:id SQL injection via :id | mitigate | parseInt + Number.isInteger guard (400 INVALID_ID) before a parameterized $1 query |
| T-02-17 | Information Disclosure | leaking private columns on public profile | mitigate | explicit SELECT of public columns only (id, display_name, avatar_url, created_at, guest_migrated_at) — never SELECT *; no email/credential/session fields |
| T-02-18 | Tampering | stored-XSS via display_name rendered on profile | mitigate | display_name already escaped at write by sanitizeDisplayName (Plan 01); CSP script-src 'self' is the second layer |
| T-02-19 | Information Disclosure | enumerating users via 404 vs 200 timing | accept | low value — opaque sequential ids; profiles are intentionally public (PROF-02); no sensitive data exposed |
</threat_model>
