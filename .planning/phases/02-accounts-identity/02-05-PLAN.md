<objective>
The Facebook OAuth sign-in vertical slice — UI -> route -> Passport -> DB -> session, end to end — mirroring the Google slice (Plan 02). A player clicks "Sign in with Facebook" on the lobby, completes the redirect flow, and returns signed-in with their guest clientId linked to a persistent account, even when Facebook withholds the email.

This reuses the entire session/Passport stack already wired by Plan 02 (sessionMiddleware, io.engine.use session-share, authRateLimit, /api/me, the linkOrPromoteAccount transaction) — it adds ONLY a second strategy, a second pair of routes, and a second button. No schema change (the provider-generic credentials model from Phase 1 already supports type='facebook' — D-14). No new linking logic (linkOrPromoteAccount is generalized to take a provider parameter in this plan).

Purpose: Delivers AUTH-05 (a third additive sign-in method) with the same SEC-05 state + session-regeneration hardening as Google, dedup by provider id (not email, per D-20), and atomic guest linking (D-06/D-07) — without regressing guest play or the Google flow.
Output: package.json (passport-facebook), server.js (FB strategy + routes), public/app.jsx (FB button + EN/VI strings), public/style.css (.facebook-signin).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> /auth/facebook callback | cross-site top-level redirect from Facebook; carries the state nonce in the session cookie |
| FacebookStrategy verify callback -> DB | profile.id/displayName from the FB token (semi-trusted); name sanitized in linkOrPromoteAccount; email never trusted as identity |
| npm registry -> node_modules | passport-facebook crosses a supply-chain boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-21 | Spoofing | Facebook OAuth callback (CSRF / state forgery) | mitigate | FacebookStrategy state:true generates + validates a per-flow nonce; forged/missing state rejected (SEC-05 / D-05) |
| T-02-22 | Elevation of Privilege | session fixation post-FB-login | mitigate | Passport 0.6+ calls req.session.regenerate() automatically on req.logIn (inherited from Plan 01's passport>=0.6 gate) |
| T-02-23 | Spoofing | trusting FB email as identity key (account takeover via spoofed/absent email) | mitigate | dedup strictly on (type='facebook', external_id=profile.id); email ignored entirely (D-16/D-20) |
| T-02-24 | Information Disclosure | Facebook access_token leak | mitigate | verify callback never stores accessToken; serializeUser writes only users.id (unchanged) |
| T-02-25 | Denial of Service | FB auth-route brute force | mitigate | authRateLimit (existing RateLimiterMemory 10/60s per IP) applied to /auth/facebook + callback |
| T-02-26 | Elevation of Privilege | FB session not sign-out-all-able (unstamped user_id) | mitigate | onFacebookCallbackSuccess stamps req.session.user_id before req.session.save (mirrors Plan 03) so the indexed DELETE matches FB rows |
| T-02-SC | Tampering | npm install (passport-facebook) | mitigate | blocking-human legitimacy checkpoint (Task 1) before install — passport-facebook is [ASSUMED] (slopcheck unavailable) |
</threat_model>
