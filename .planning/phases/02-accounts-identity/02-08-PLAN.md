<objective>
The async email-verification slice (AUTH-07, D-19): a gracefully-degrading Resend mailer module, a verification email fired non-blocking after email signup, and a GET /auth/verify endpoint that flips email_verified via the single-use token from Plan 06. Play is never gated on verification — signup logs the user in immediately (Plan 07) and the email is best-effort.

This adds mailer.js (the project's first email-sending capability) following the established optional-feature pattern: like Redis/audio/localStorage, it no-ops + logs when RESEND_API_KEY is unset and never throws, so signup and play work with or without email configured (D-18). It uses createAuthToken('verify', 24h) and consumeAuthToken from Plan 06 — no new token logic.

Purpose: AUTH-07 closes the email-account loop (verify ownership) without sacrificing the guest-first low-friction ethos (D-19): verification is additive, async, and non-blocking. The verification-link endpoint is a tampering surface, so single-use + expiry (already in Plan 06's token helpers) plus a clean BAD_TOKEN rejection are the security core.
Output: package.json (resend), mailer.js, server.js (async send + /auth/verify), public/app.jsx (verify result + unverified hint + EN/VI), test/auth.test.js (AUTH-07 assertions).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| emailed link -> GET /auth/verify?token | token from an email crosses back as untrusted query input; must be single-use, expiring, unguessable |
| server -> Resend API | outbound email; API key is a secret; absence must degrade gracefully not crash |
| npm registry -> node_modules | resend crosses a supply-chain boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-39 | Tampering | verification-link token forgery/replay | mitigate | consumeAuthToken enforces single-use (conditional consumed_at UPDATE) + expiry (24h); forged/expired/reused -> BAD_TOKEN -> verifyError, no flip |
| T-02-40 | Information Disclosure | RESEND_API_KEY / MAIL_FROM leak | mitigate | key read from env only, never logged or returned; mailer logs only the recipient + skip/failure state |
| T-02-41 | Denial of Service | signup blocked when email provider is down | mitigate | send is non-blocking + best-effort; mailer catches all errors and returns {error}; signup response never waits on or fails from the send (D-19) |
| T-02-42 | Denial of Service | unconfigured email provider breaks signup | mitigate | mailer no-ops + logs when RESEND_API_KEY unset (graceful degradation, store.js pattern) — signup/play unaffected (D-18) |
| T-02-43 | Spoofing | verifying an account the requester does not own | mitigate | token is bound to user_id at creation (createAuthToken) and only that user_id is verified on consume; token entropy 256-bit (Plan 06) |
| T-02-SC | Tampering | npm install (resend) | mitigate | blocking-human legitimacy checkpoint (Task 1) before install — resend is [ASSUMED] (slopcheck unavailable) |
</threat_model>
