<objective>
The email-account persistence foundation: a NEW numbered migration (003_email_accounts.sql) and the db.js functions the email/verification/reset slices build on. This is the heavy email slice's substrate — it does NOT wire any Express route or UI (those are Plans 07/08/09). It produces unit-testable, bcrypt-backed account-creation + login + token functions, and extends the schema WITHOUT touching the FINAL 002_accounts.sql.

Why a new migration: 002_accounts.sql is FINAL and Google-only (it has no password_hash, email, email_verified, or token storage). Per D-14/D-15, all new email-account schema lands in 003_email_accounts.sql: email + email_verified on users, a nullable password_hash on credentials (for type='email'), and a dedicated auth_tokens table for single-use, time-limited verification + reset tokens (token-storage shape is the planner's call per D-15 — a table is chosen for clean single-use + expiry + multi-purpose semantics).

Purpose: Email/password is the highest-value-threat method (password storage, brute force, token tampering). Getting the bcrypt hashing, the no-enumeration login compare, and the single-use/expiry token semantics correct and unit-tested in isolation — before any route exists — is the AUTH-06 linchpin, the same way Plan 01's linkOrPromoteAccount de-risked OAuth.
Output: package.json (bcrypt), migrations/003_email_accounts.sql, db.js (+5 exports), test/auth.test.js (AUTH-06 assertions).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client email+password -> createEmailAccount/verifyEmailLogin | fully untrusted input; password must be hashed at rest, login must not leak account existence |
| auth_tokens.token -> consumeAuthToken | token arrives later from an emailed link (Plans 07-09); must be single-use, expiring, and unguessable |
| npm registry -> node_modules | bcryptjs crosses a supply-chain boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-27 | Information Disclosure | password storage | mitigate | bcrypt hash at cost >= 10 (bcryptjs); plaintext never stored or logged; test asserts stored hash != plaintext |
| T-02-28 | Spoofing | account enumeration via login response | mitigate | verifyEmailLogin returns identical {error:'AUTH_FAILED'} for unknown-email and wrong-password (no distinguishing message/timing branch) |
| T-02-29 | Tampering | weak passwords | mitigate | min 8 chars enforced server-side -> WEAK_PASSWORD (D-17); enforcement is in db.js, not just UI |
| T-02-30 | Spoofing | guessable / reusable tokens | mitigate | crypto.randomBytes(32) hex token (256-bit), UNIQUE constraint, single-use via conditional consumed_at UPDATE, expiry via expires_at |
| T-02-31 | Tampering | SQL injection via email/token | mitigate | all auth_tokens + credential SQL parameterized ($1..); email normalized but still bound, never concatenated |
| T-02-32 | Elevation of Privilege | duplicate email account bypassing dedup | mitigate | EMAIL_IN_USE guard + credentials UNIQUE(type='email', external_id=normalized email); D-20 keeps providers distinct |
| T-02-SC | Tampering | npm install (bcryptjs) | mitigate | blocking-human legitimacy checkpoint (Task 1) before install — bcryptjs is [ASSUMED] (slopcheck unavailable) |
</threat_model>
