<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google token -> linkOrPromoteAccount | sub/name/avatarUrl originate from the Google ID token (semi-trusted); name is rendered later, must be sanitized |
| session -> pendingClientId -> SQL | pendingClientId comes from server-side session only (set in Plan 02); never from a request body — but must still be parameterized |
| npm registry -> node_modules | new packages cross a supply-chain boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | display_name stored via linkOrPromoteAccount | mitigate | sanitizeDisplayName(name) = escapeHtml + control-char strip + slice(0,40) at write time (D-09, V5) |
| T-02-02 | Tampering | all link SQL | mitigate | parameterized $1/$2 binding only; no string concat (db.js convention) |
| T-02-03 | Spoofing | dedup key for Google identity | mitigate | dedup on credentials(type,external_id)=('google',sub); sub is stable, email is not |
| T-02-04 | Elevation of Privilege | duplicate/orphaned users rows (Pitfall 1) | mitigate | single BEGIN/COMMIT transaction; D-06 promotes existing guest row, D-07 adopts credential — never blind INSERT users |
| T-02-SC | Tampering | npm installs (passport, passport-google-oauth20, express-session, connect-pg-simple) | mitigate | blocking-human legitimacy checkpoint (Task 1) before install — all four are [ASSUMED] (slopcheck unavailable) |
</threat_model>
