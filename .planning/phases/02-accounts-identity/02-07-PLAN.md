<objective>
The email/password sign-in vertical slice — collapsible UI form -> rate-limited routes -> db helpers -> regenerated session, end to end. A player expands "or continue with email" (D-21), signs up or logs in, and lands signed-in exactly like an OAuth user, with their guest clientId linked.

This wires Plan 06's createEmailAccount / verifyEmailLogin into live Express routes with the same session hygiene as OAuth: explicit req.session.regenerate() (email login has no Passport strategy doing it automatically, so it is done manually — D-05/SEC-05), req.login to populate req.user, and the same user_id stamp Plan 03 relies on for sign-out-all. It extends the Phase-1 rate limiter to /auth/login + /auth/signup (D-17) and adds the collapsible email form beneath the Google + Facebook buttons (D-21).

Purpose: Delivers AUTH-06 (the third sign-in method) end to end. Email login is the one path where session fixation is NOT auto-handled by Passport's OAuth strategy, so the manual regenerate + login + stamp ordering is the security-critical core of this slice.
Output: server.js (signup/login routes), public/app.jsx (EmailAuthForm + EN/VI strings), public/style.css (.email-auth-*), test/auth.test.js (AUTH-06 route assertions).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /auth/signup,/auth/login | untrusted email+password JSON; password hashed in db layer, login must not enumerate, routes rate-limited |
| email login -> session | no Passport OAuth strategy here, so session.regenerate must be called manually before req.login (fixation defense) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-33 | Elevation of Privilege | session fixation on email login (no auto-regenerate) | mitigate | explicit req.session.regenerate() BEFORE req.login on both /auth/signup and /auth/login (SEC-05/D-05 parity with OAuth) |
| T-02-34 | Denial of Service | login/signup brute force | mitigate | authRateLimit (RateLimiterMemory 10/60s per IP) on both routes -> 429 RATE_LIMITED (D-17) |
| T-02-35 | Spoofing | account enumeration via login/signup responses | mitigate | login -> uniform 401 AUTH_FAILED; signup EMAIL_IN_USE is unavoidable for usability but rate-limited to blunt enumeration; no timing branch beyond bcrypt |
| T-02-36 | Elevation of Privilege | email session not sign-out-all-able | mitigate | req.session.user_id stamped before req.session.save (mirrors Plan 03) so the indexed DELETE covers email sessions |
| T-02-37 | Tampering | injected email/password into SQL | mitigate | all DB access via Plan 06 parameterized helpers; routes never build SQL |
| T-02-38 | Information Disclosure | password echoed in logs/response | mitigate | routes never log req.body; responses return only {id,displayName,avatarUrl}; bcrypt hashing stays in db layer |
</threat_model>
