<objective>
Implement the backend slice for match history: a paginated API endpoint `GET /api/matches` with offset pagination and dynamic filters (result, mode, wager), plus upgrade the existing `/api/profile/:userId` endpoint to return real win/loss statistics from the matches table instead of hardcoded zeros.

This provides the data layer that the frontend (Plan 02) consumes. After this plan lands, the API is fully functional and testable independently of the UI.

Purpose: The matches table already stores all completed games (Phase 3). This plan makes that data accessible to the client through a clean REST API with the same auth pattern used by `/api/wallet`.
Output: `db.js` additions (getMatchHistory, getUserStats), `server.js` route additions, `test/match-history.test.js`.
</objective>

  <action>
    Add two new exported functions to db.js:

    **`getMatchHistory(userId, { result, mode, wager }, page, limit)`**
    - Returns `{ matches: [...], total, page, hasMore }`
    - Uses a single parameterized query with CASE expressions:
      - `result` column: CASE WHEN winner_id = $1 THEN 'win' ELSE 'loss'
      - `opponent_id`: CASE WHEN winner_id = $1 THEN loser_id ELSE winner_id
      - `points_delta`: CASE WHEN winner_id = $1 THEN floor(stake*2*0.9) ELSE -stake
    - JOIN users u ON u.id = opponent_id (for display_name, avatar_url)
    - Dynamic WHERE filters using parameterized 'all' checks:
      - result filter: ($2 = 'all' OR ($2 = 'win' AND winner_id = $1) OR ($2 = 'loss' AND loser_id = $1))
      - mode filter: ($3 = 'all' OR m.mode = $3)
      - wager filter: ($4 = 'all' OR ($4 = 'has' AND m.stake > 0) OR ($4 = 'none' AND m.stake = 0))
    - ORDER BY m.ended_at DESC, LIMIT $5 OFFSET $6
    - Separate COUNT query for total (same WHERE clause)
    - Cap limit to 50 max, default 20
    - Graceful degradation: if DATABASE_URL/PGHOST not set, return empty result

    **`getUserStats(userId)`**
    - Returns `{ wins, losses, gamesPlayed, winRate }`
    - Query: SELECT COUNT(*) total, COUNT(*) FILTER (WHERE winner_id = $1) AS wins FROM matches WHERE winner_id = $1 OR loser_id = $1
    - Compute: losses = total - wins, winRate = total > 0 ? Math.round(wins/total*100) : 0
    - Graceful degradation: if no DB, return { wins: 0, losses: 0, gamesPlayed: 0, winRate: 0 }

    Add both to module.exports alongside existing exports.
  </action>

    Add Express route `GET /api/matches` in server.js (near other /api/* routes):

    ```javascript
    app.get("/api/matches", async (req, res) => {
      if (!req.user?.id) return res.status(401).json({ error: "Unauthorized" });
      const page = Math.max(1, parseInt(req.query.page) || 1);
      const limit = Math.min(50, Math.max(1, parseInt(req.query.limit) || 20));
      const result = ["all", "win", "loss"].includes(req.query.result) ? req.query.result : "all";
      const mode = ["all", "classic", "advance"].includes(req.query.mode) ? req.query.mode : "all";
      const wager = ["all", "has", "none"].includes(req.query.wager) ? req.query.wager : "all";
      try {
        const data = await getMatchHistory(req.user.id, { result, mode, wager }, page, limit);
        res.json(data);
      } catch (e) {
        console.error("[api] /api/matches error:", e.message);
        res.status(500).json({ error: "Internal error" });
      }
    });
    ```

    Input validation:
    - page: integer >= 1 (default 1)
    - limit: integer 1-50 (default 20)
    - result/mode/wager: whitelist validation (default "all" for invalid values)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser query params → server API | Untrusted filter/page values from URL params |
| server → Postgres | All queries cross into DB |
| session → user identity | req.user.id from express-session determines whose history is returned |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering/Injection | Query params in /api/matches | mitigate | Whitelist validation for filter values; parseInt for numeric params; all SQL uses $N binding |
| T-13-02 | Information Disclosure | Match history of other users | mitigate | API always uses req.user.id from session — user cannot query another user's history |
| T-13-03 | Elevation of Privilege | Guest accessing history | mitigate | 401 returned when req.user is absent (same pattern as /api/wallet) |
| T-13-04 | Denial of Service | Large offset queries | accept | Limit capped at 50; modest data volume per user (<10K matches); existing indexes sufficient |
</threat_model>
