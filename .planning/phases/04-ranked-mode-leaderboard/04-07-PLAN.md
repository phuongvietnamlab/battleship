<objective>
Close CR-02 (RANK-04 production-hardening) from 04-VERIFICATION.md: the public, unauthenticated `GET /api/leaderboard` endpoint (server.js ~62-70) has no rate limit, and in the documented RAM-only Render deployment (no REDIS_URL) `store.getLeaderboardCache()` always returns null, so every request runs `buildLeaderboard` (a JOIN + ORDER BY rating DESC LIMIT 100 over ratings/users) against Postgres with no amortization. The phase plan explicitly scoped DDoS mitigation for the leaderboard. The codebase already has the `RateLimiterMemory` + `authRateLimit` convention (server.js ~203-216) but it is not applied to this route.

Fix two things: (1) apply an in-process per-IP rate limiter to the route, mirroring the existing `authRateLimit` middleware shape; (2) add a short-TTL in-process (RAM) cache for the leaderboard payload so RAM-only mode amortizes repeated reads. Both are gracefully degradable and add no new dependency (rate-limiter-flexible is already required at server.js ~203).

Purpose: RANK-04 production-hardening — the leaderboard endpoint resists an unauthenticated request flood and does not hammer Postgres in RAM-only mode.
Output: a `leaderboardLimiter` + `leaderboardRateLimit` middleware on GET /api/leaderboard; an in-process short-TTL leaderboard cache fallback; tests asserting 429-after-burst and cache amortization.
