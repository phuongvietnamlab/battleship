# Plan 06: Operational controls API

## Objective
Implement server health monitoring, season management trigger, maintenance mode toggle, runtime config editor, and backup trigger endpoints.

## must_haves
- [ ] Health endpoint returns memory, event loop lag, pg pool stats, Redis status, uptime
- [ ] Season reset triggerable from admin API with confirmation
- [ ] Maintenance mode toggle blocks new connections while allowing existing games to finish
- [ ] Config editor reads/writes runtime_config table
- [ ] Backup trigger executes pg_dump with rate limiting (max 1/hour)

## Tasks

### Task 1: Server health endpoint

<read_first>
- admin-api.js
- server.js
- store.js
</read_first>

<action>
In server.js, at module scope (near the top, after imports), add event loop monitoring:

1. Import monitorEventLoopDelay from "perf_hooks":
   const { monitorEventLoopDelay } = require("perf_hooks");
   const eventLoopHistogram = monitorEventLoopDelay({ resolution: 20 });
   eventLoopHistogram.enable();

In admin-api.js, add GET "/ops/health" with requireRole("moderator"):

1. Access pool (from db.js), store (from store.js), io, rooms, and eventLoopHistogram (passed via mountAdminRoutes params or require)
2. Build health response:
   {
     uptime: Math.floor(process.uptime()),
     memory: { rss, heapUsed, heapTotal, external } from process.memoryUsage() (convert to MB: Math.round(bytes / 1024 / 1024)),
     eventLoop: { min, max, mean, p99 } from eventLoopHistogram (divide by 1e6 to get ms, round to 2 decimals),
     pgPool: { total: pool.totalCount, idle: pool.idleCount, waiting: pool.waitingCount },
     redis: { connected: store.isEnabled() },
     rooms: Object.keys(rooms).length,
     onlinePlayers: io.of("/").sockets.size,
     nodeVersion: process.version,
     pid: process.pid
   }
3. Return 200 with the health object

Note: server.js must export or pass eventLoopHistogram to admin-api.js. Simplest: pass as parameter to mountAdminRoutes(router, io, rooms, eventLoopHistogram).
</action>

<acceptance_criteria>
- GET /api/admin/ops/health returns all health metrics as JSON
- Memory values are in MB (rounded integers)
- Event loop values are in milliseconds (2 decimal places)
- pgPool shows totalCount, idleCount, waitingCount from the pg Pool instance
- Redis status shows connected boolean
- rooms count reflects in-memory room map size
- Endpoint accessible to all admin roles (moderator+)
</acceptance_criteria>

### Task 2: Season management endpoint

<read_first>
- admin-api.js
- scripts/season-reset.js
- db.js
</read_first>

<action>
Add two endpoints with requireRole("super_admin"):

GET "/ops/season/preview":
1. Query count of affected users: SELECT COUNT(*) FROM ratings
2. Query current season info from runtime_config WHERE key='current_season' (or return default season 1 if not set)
3. Return { affectedUsers, currentSeason, nextSeason: currentSeason + 1 }

POST "/ops/season/reset":
1. Accept body: confirm (required, must be the string "RESET_SEASON" for safety)
2. If confirm !== "RESET_SEASON", return 400 { error: "CONFIRMATION_REQUIRED" }
3. Execute the season reset logic (same as scripts/season-reset.js):
   - INSERT INTO rating_history SELECT * FROM ratings (archive current ratings)
   - UPDATE ratings SET rating = 1500 + (rating - 1500) * 0.5, rd = 350, volatility = 0.06, games_played = 0
   - UPSERT runtime_config SET key='current_season', value=nextSeason
4. Call logAdminAction with action "ops.season_reset", details { affectedUsers, newSeason }
5. Return 200 { ok: true, affectedUsers, newSeason }

If the ratings table doesn't exist (Phase 4 may have been removed per roadmap), make the endpoint return 400 { error: "RATINGS_NOT_AVAILABLE" } gracefully.
</action>

<acceptance_criteria>
- GET /api/admin/ops/season/preview returns count of affected users and current season
- POST /api/admin/ops/season/reset with { confirm: "RESET_SEASON" } performs the reset
- POST without correct confirm string returns 400 CONFIRMATION_REQUIRED
- Only super_admin can trigger (admin/moderator get 403)
- Action is logged to admin_audit_log with affected user count
- If ratings table doesn't exist, returns graceful error (not 500)
</acceptance_criteria>

### Task 3: Maintenance mode toggle

<read_first>
- admin-api.js
- server.js
</read_first>

<action>
In server.js, add maintenance mode infrastructure:

1. At module scope, add: let maintenanceMode = false;
2. After pool is available, load initial state from DB:
   pool.query("SELECT value FROM runtime_config WHERE key='maintenance_mode'").then(({rows}) => {
     if (rows.length > 0) maintenanceMode = rows[0].value === true || rows[0].value === 'true';
   }).catch(() => {});

3. Add maintenance check middleware BEFORE the game Socket.IO connection handler (in io.use):
   Add to the existing io.use middleware chain (or add new one):
   If maintenanceMode is true AND this is NOT an admin connection (check if path is /admin):
   - Emit "maintenance" event with message
   - Return next(new Error("MAINTENANCE_MODE"))

4. Also add Express middleware for non-admin routes:
   app.use((req, res, next) => {
     if (maintenanceMode && !req.path.startsWith("/admin") && !req.path.startsWith("/api/admin") && req.path !== "/healthz") {
       return res.status(503).json({ error: "MAINTENANCE_MODE", message: "Server is under maintenance" });
     }
     next();
   });
   Place this AFTER admin routes but BEFORE game static files.

In admin-api.js, add:

GET "/ops/maintenance" with requireRole("moderator"):
1. Return { enabled: maintenanceMode } (access via passed reference or getter function)

POST "/ops/maintenance" with requireRole("super_admin"):
1. Accept body: enabled (required boolean)
2. Update in-memory: maintenanceMode = enabled
3. Upsert DB: INSERT INTO runtime_config (key, value, updated_by) VALUES ('maintenance_mode', $1::jsonb, $2) ON CONFLICT (key) DO UPDATE SET value=$1::jsonb, updated_by=$2, updated_at=now()
4. Emit to admin namespace: adminNs.emit("maintenance", { enabled })
5. Call logAdminAction with action enabled ? "ops.maintenance_on" : "ops.maintenance_off"
6. Return 200 { ok: true, enabled }

Pass maintenanceMode reference to admin-api.js via a getter/setter pattern or module-level variable. Simplest: export a getMaintenanceMode/setMaintenanceMode pair from server.js or share via the mountAdminRoutes options object.
</action>

<acceptance_criteria>
- GET /api/admin/ops/maintenance returns current maintenance mode state
- POST /api/admin/ops/maintenance with { enabled: true } activates maintenance mode
- When maintenance is active, new Socket.IO connections from players are rejected
- When maintenance is active, HTTP requests to non-admin routes return 503
- Admin routes (/admin/*, /api/admin/*) continue working during maintenance
- /healthz continues working during maintenance (for uptime monitors)
- Maintenance state persists across server restarts (stored in runtime_config)
- State is loaded from DB on boot
- Admin WebSocket receives "maintenance" event when toggled
</acceptance_criteria>

### Task 4: Config editor and backup trigger

<read_first>
- admin-api.js
- db.js
</read_first>

<action>
Add config and backup endpoints:

GET "/ops/config" with requireRole("admin"):
1. SELECT key, value, description, updated_at FROM runtime_config ORDER BY key
2. Return { configs: [{ key, value, description, updatedAt }, ...] }

PUT "/ops/config/:key" with requireRole("super_admin"):
1. Accept body: value (required, any JSON-serializable value), description (optional string)
2. Validate key matches /^[a-z_]+$/ pattern (alphanumeric + underscores only)
3. UPSERT: INSERT INTO runtime_config (key, value, description, updated_by) VALUES ($1, $2::jsonb, $3, $4) ON CONFLICT (key) DO UPDATE SET value=$2::jsonb, description=COALESCE($3, runtime_config.description), updated_by=$4, updated_at=now()
4. Call logAdminAction with action "ops.config_update", details { key, value }
5. Return 200 { ok: true }

POST "/ops/backup" with requireRole("super_admin"):
1. Check rate limit: read runtime_config WHERE key='last_backup_at'. If value is within the last hour, return 429 { error: "BACKUP_RATE_LIMITED", lastBackup: value, nextAllowedAt }
2. Execute pg_dump via child_process.execFile:
   - Command: "pg_dump"
   - Args: ["--format=custom", "--file=" + backupPath] where backupPath = path.join("/tmp", "backup-" + new Date().toISOString().replace(/[:.]/g, "-") + ".dump")
   - Env: inherit process.env (PGHOST/PGUSER/PGPASSWORD/PGDATABASE are already set)
   - Timeout: 120000 ms
   - Do NOT pass password in command args (use env)
3. On success: UPSERT runtime_config key='last_backup_at' with ISO timestamp value, and key='last_backup_path' with the file path
4. Call logAdminAction with action "ops.backup_trigger", details { path: backupPath, durationMs }
5. Return 200 { ok: true, path: backupPath, timestamp: new Date().toISOString() }
6. On failure: return 500 { error: "BACKUP_FAILED", message: stderr_output }

GET "/ops/backup" with requireRole("admin"):
1. Read runtime_config WHERE key IN ('last_backup_at', 'last_backup_path')
2. Return { lastBackupAt, lastBackupPath } or { lastBackupAt: null } if never run
</action>

<acceptance_criteria>
- GET /api/admin/ops/config returns all runtime config entries
- PUT /api/admin/ops/config/some_key upserts the value (only super_admin)
- Key validation rejects keys with special characters
- POST /api/admin/ops/backup triggers pg_dump and returns path on success
- Backup is rate-limited to once per hour (429 if too soon)
- Backup timeout is 120 seconds
- Password is passed via environment, never in command args
- GET /api/admin/ops/backup returns last backup metadata
- All operations are audit-logged
</acceptance_criteria>
