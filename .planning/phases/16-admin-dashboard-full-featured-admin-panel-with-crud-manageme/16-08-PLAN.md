# Plan 08: Admin frontend views

## Objective
Build all admin page views — login form, analytics dashboard with charts and real-time counters, user management list/detail, match management, content management, moderation queue, and operations panel — with full responsive design.

## must_haves
- [ ] Login page with email/password form and error handling
- [ ] Dashboard with overview cards, 3 charts (users, matches, points), and live stats panel
- [ ] User list with DataTable, search, detail panel, ban/edit actions
- [ ] Match list with filters, detail view, void action
- [ ] Content management views (emojis, announcements)
- [ ] Moderation view (reports queue, chat log viewer)
- [ ] Operations view (health metrics, maintenance toggle, config, backup)
- [ ] Responsive: full experience desktop/tablet, read-only dashboard on mobile

## Tasks

<action>
Create public/admin/pages/Login.jsx:

1. Render a centered login card with:
   - Title: "Battleship Admin" with ship icon/emoji
   - Email input (type="email", required, autocomplete="email")
   - Password input (type="password", required, autocomplete="current-password")
   - Submit button: "Sign In"
   - Error message display area (shown when login fails)

2. On form submit:
   - Prevent default
   - Set loading state
   - Call POST /api/admin/login with { email, password } via useApi().post
   - On success: call setAdminUser(response.user), navigate to #/dashboard
   - On error: display error message (AUTH_FAILED → "Invalid credentials", NOT_ADMIN → "No admin access", RATE_LIMITED → "Too many attempts, try later")

3. Styling: card centered vertically and horizontally on page, max-width 400px, dark-themed card background. Input fields styled with --bg-secondary, --border, --text-primary variables.

4. Accessibility: form has aria-label, inputs have labels, error has role="alert"
</action>

<acceptance_criteria>
- Login page renders at #/login with email and password fields
- Submitting valid admin credentials navigates to #/dashboard
- Invalid credentials show "Invalid credentials" error message
- Non-admin user login shows "No admin access" message
- Rate-limited response shows "Too many attempts" message
- Form is accessible: labeled inputs, alert role on error
- Loading state disables submit button and shows spinner
</acceptance_criteria>

</read_first>

<action>
Create public/admin/pages/Users.jsx:

1. User list view (default):
   - Search input (debounced 300ms) that filters by name/email/ID
   - Status filter dropdown: All, Active, Banned, Deleted
   - DataTable with columns: ID, Display Name, Email, Created At, Status (badge), Actions
   - Status badge: green "Active", red "Banned", gray "Deleted"
   - Actions column: "View" button opens detail panel
   - Bulk actions when rows selected: "Ban Selected", "Export Selected"
   - CSV export button calls GET /api/admin/users/export with current filters

2. User detail panel (slide-in from right or modal):
   - Opens when clicking "View" on a user row
   - Shows: avatar, display_name, email, created_at, auth methods list, wallet balance
   - Tabs: Profile, Match History (last 20 matches), Transactions (last 50), Ban History
   - Action buttons (based on role):
     - Edit (opens inline form for display_name)
     - Ban/Unban (opens confirm dialog with reason + duration inputs)
     - Mute/Unmute (similar to ban)
     - Adjust Points (opens form with amount + reason)
     - Delete (opens typed confirmation dialog, must type "DELETE")
   - All actions use useConfirm() for destructive operations and useToast() for success/error feedback

3. Pagination: DataTable handles page changes via onPageChange callback, which refetches GET /api/admin/users?page=N

4. Sorting: clicking column headers updates sort params and refetches
</action>

<acceptance_criteria>
- Users page shows paginated user list with search and status filter
- Searching by partial name returns matching results (debounced)
- Clicking "View" opens user detail with profile, matches, transactions, bans tabs
- Ban action opens confirmation dialog requiring reason; on confirm calls POST /users/:id/ban
- Point adjustment prompts for amount and reason
- Delete requires typing "DELETE" in confirmation
- CSV export downloads file with current filter applied
- All actions show success/error toast
- DataTable pagination and sorting work correctly
</acceptance_criteria>
