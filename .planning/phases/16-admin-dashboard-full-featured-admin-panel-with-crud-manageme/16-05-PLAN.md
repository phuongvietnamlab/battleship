# Plan 05: Analytics API

## Objective
Implement analytics endpoints for dashboard overview cards, time-series chart data (user growth, match activity, points economy), retention cohorts, revenue metrics, real-time stats via Socket.IO /admin namespace, and daily stats aggregation.

## must_haves
- [ ] Dashboard overview endpoint with key metrics (users, DAU, matches today, points, online count)
- [ ] Time-series endpoints for user growth, match activity, and points economy charts
- [ ] Retention cohort computation (D1/D7/D30)
- [ ] Revenue metrics with conversion rate
- [ ] Real-time WebSocket stats push via /admin namespace
- [ ] Daily stats aggregation running on setInterval

### Task 2: Time-series chart data endpoints

<read_first>
- admin-api.js
- db.js
</read_first>

<action>
Add three chart data endpoints, all with requireRole("moderator"):

GET "/analytics/users" (user growth chart — ADM-09):
1. Accept query: range (default "30", options: "30", "90", "365")
2. Query daily_stats table: SELECT date, new_users FROM daily_stats WHERE date >= CURRENT_DATE - ($1 || ' days')::interval ORDER BY date ASC
3. If daily_stats is empty for the range, fall back to live query: SELECT created_at::date AS date, COUNT(*) AS new_users FROM users WHERE created_at >= CURRENT_DATE - ($1 || ' days')::interval GROUP BY created_at::date ORDER BY date ASC
4. Return { data: [{ date: "YYYY-MM-DD", newUsers: N }, ...] }

GET "/analytics/matches" (match activity chart — ADM-10):
1. Accept query: range (default "30")
2. Query daily_stats: SELECT date, matches_played, matches_classic, matches_wagered FROM daily_stats WHERE date >= CURRENT_DATE - ($1 || ' days')::interval ORDER BY date ASC
3. Fallback live query if daily_stats empty: SELECT ended_at::date AS date, COUNT(*) AS matches_played, COUNT(*) FILTER (WHERE stake = 0) AS matches_classic, COUNT(*) FILTER (WHERE stake > 0) AS matches_wagered FROM matches WHERE ended_at >= CURRENT_DATE - ($1 || ' days')::interval GROUP BY ended_at::date ORDER BY date ASC
4. Return { data: [{ date, matchesPlayed, matchesClassic, matchesWagered }, ...] }

GET "/analytics/points" (points economy chart — ADM-11):
1. Accept query: range (default "30")
2. Query daily_stats: SELECT date, points_earned, points_spent FROM daily_stats WHERE date >= CURRENT_DATE - ($1 || ' days')::interval ORDER BY date ASC
3. Fallback: SELECT created_at::date AS date, COALESCE(SUM(amount) FILTER (WHERE amount > 0), 0) AS points_earned, COALESCE(ABS(SUM(amount) FILTER (WHERE amount < 0)), 0) AS points_spent FROM transactions WHERE created_at >= CURRENT_DATE - ($1 || ' days')::interval GROUP BY created_at::date ORDER BY date ASC
4. Return { data: [{ date, pointsEarned, pointsSpent }, ...] }
</action>

<acceptance_criteria>
- GET /api/admin/analytics/users?range=30 returns up to 30 data points with date and newUsers
- GET /api/admin/analytics/matches?range=90 returns match activity split by classic/wagered
- GET /api/admin/analytics/points?range=365 returns points earned vs spent per day
- Range parameter is validated: only "30", "90", "365" accepted; default to "30" for unknown values
- Endpoints gracefully fall back to live queries when daily_stats table is empty
- Data is sorted chronologically (oldest first) for chart rendering
</acceptance_criteria>

### Task 3: Retention and revenue metrics endpoints

<read_first>
- admin-api.js
- db.js
</read_first>

<action>
Add two endpoints with requireRole("moderator"):

GET "/analytics/retention" (ADM-14):
1. Compute retention cohorts for the last 30 days (each day is a cohort)
2. For each cohort date, compute:
   - cohort_size: COUNT of users created on that date
   - d1_retained: users from that cohort who played a match on date+1
   - d7_retained: users from that cohort who played a match on date+7
   - d30_retained: users from that cohort who played a match on date+30
3. SQL approach: WITH cohort AS (SELECT id, created_at::date AS cohort_date FROM users WHERE created_at >= CURRENT_DATE - INTERVAL '60 days') — then for each retention day, count distinct users from cohort who appear in matches on the target date.
4. Limit to last 30 cohort dates for response size. Group results by week for summary.
5. Return { cohorts: [{ date, cohortSize, d1: { retained, pct }, d7: { retained, pct }, d30: { retained, pct } }, ...], summary: { avgD1, avgD7, avgD30 } }

GET "/analytics/revenue" (ADM-12):
1. Total points spent (all time): SELECT COALESCE(ABS(SUM(amount)), 0) FROM transactions WHERE amount < 0
2. Top spenders (top 10): SELECT t.user_id, u.display_name, ABS(SUM(t.amount)) as total_spent FROM transactions t JOIN users u ON u.id = t.user_id WHERE t.amount < 0 GROUP BY t.user_id, u.display_name ORDER BY total_spent DESC LIMIT 10
3. Conversion rate: SELECT COUNT(*) FILTER (WHERE guest_migrated_at IS NOT NULL) as converted, COUNT(*) as total FROM users
4. Points spent by category: SELECT type, ABS(SUM(amount)) as total FROM transactions WHERE amount < 0 GROUP BY type ORDER BY total DESC
5. Return { totalPointsSpent, topSpenders: [...], conversionRate: { converted, total, pct }, spendingByCategory: [...] }
</action>

<acceptance_criteria>
- GET /api/admin/analytics/retention returns cohort data with D1/D7/D30 retention percentages
- Summary averages are computed across all cohorts
- GET /api/admin/analytics/revenue returns total spend, top 10 spenders, conversion rate
- Conversion rate percentage is rounded to 1 decimal place
- Retention query handles cohorts with 0 users (no division by zero)
- Empty data returns zeros, not errors
</acceptance_criteria>

### Task 4: Real-time stats via Socket.IO /admin namespace

<read_first>
- server.js
- admin-auth.js
</read_first>

<action>
In server.js, after the existing io setup and admin mounting, add the admin WebSocket namespace:

1. Create adminNs = io.of("/admin")

2. Add auth middleware to adminNs:
   adminNs.use(async (socket, next) => {
     Parse cookie from socket.handshake.headers.cookie to extract "admin.sid" value (using a simple regex or cookie-parse logic).
     If no sid cookie found, return next(new Error("ADMIN_AUTH_REQUIRED")).
     Query admin_sessions table: SELECT sess FROM admin_sessions WHERE sid=$1 AND expire > now().
     If no row, return next(new Error("SESSION_EXPIRED")).
     Parse sess JSON, extract adminUserId. Query admin_roles for active role.
     If no active role, return next(new Error("NOT_ADMIN")).
     Set socket.data.adminUser = { id: adminUserId, role }.
     Call next().
   })

3. Add fast stats push (every 5 seconds):
   setInterval(() => {
     if (adminNs.sockets.size === 0) return;
     adminNs.emit("stats", {
       online: io.of("/").sockets.size,
       activeMatches: Object.values(rooms).filter(r => r.started).length,
       memory: process.memoryUsage(),
       uptime: Math.floor(process.uptime()),
       ts: Date.now()
     });
   }, 5000).unref();

4. Add slow stats push (every 60 seconds):
   setInterval(async () => {
     if (adminNs.sockets.size === 0) return;
     try {
       const pgStats = { total: pool.totalCount, idle: pool.idleCount, waiting: pool.waitingCount };
       adminNs.emit("slowStats", { pgPool: pgStats, ts: Date.now() });
     } catch (e) { console.error("[admin-ws] slow stats error:", e.message); }
   }, 60000).unref();

5. Log admin namespace connections: adminNs.on("connection", (socket) => { console.log("[admin-ws] connected:", socket.data.adminUser.id); })
</action>

<acceptance_criteria>
- Socket.IO /admin namespace exists and requires admin session cookie for connection
- Unauthenticated WebSocket connections to /admin are rejected with error
- Connected admin clients receive "stats" event every 5 seconds with online count, active matches, memory, uptime
- Connected admin clients receive "slowStats" event every 60 seconds with pg pool stats
- Stats emission is skipped when no admin clients are connected (efficiency)
- Both intervals use .unref() to not keep the process alive
</acceptance_criteria>

### Task 5: Daily stats aggregation scheduler

<read_first>
- server.js
- db.js
</read_first>

<action>
In server.js, after the admin namespace setup, add the daily stats aggregation:

1. Define async function computeDailyStats(dateStr) that:
   - Accepts a date string in 'YYYY-MM-DD' format
   - Runs the INSERT INTO daily_stats ... ON CONFLICT (date) DO UPDATE query from the research doc
   - Computes: new_users (users created on date), active_users (distinct users in matches on date), matches_played, matches_classic (stake=0), matches_wagered (stake>0), points_earned (positive transactions), points_spent (abs of negative transactions), points_wagered (sum of stakes*2 for wagered matches)
   - All using parameterized queries with the date as $1

2. Set up hourly aggregation timer:
   setInterval(async () => {
     try {
       const yesterday = new Date(Date.now() - 86400000).toISOString().split("T")[0];
       const today = new Date().toISOString().split("T")[0];
       await computeDailyStats(yesterday);
       await computeDailyStats(today);
     } catch (e) { console.error("[stats] aggregation failed:", e.message); }
   }, 60 * 60 * 1000).unref();

3. Run initial aggregation 30 seconds after boot:
   setTimeout(async () => {
     try {
       const today = new Date().toISOString().split("T")[0];
       await computeDailyStats(today);
     } catch (e) { console.error("[stats] initial aggregation failed:", e.message); }
   }, 30000).unref();
</action>

<acceptance_criteria>
- computeDailyStats function computes and upserts daily metrics for a given date
- Hourly timer recomputes yesterday and today stats (handles day boundary)
- Initial aggregation runs 30 seconds after server boot
- ON CONFLICT DO UPDATE ensures idempotent re-runs
- Timer uses .unref() matching existing project pattern (sweepRooms)
- Errors are caught and logged, never crash the server
</acceptance_criteria>
