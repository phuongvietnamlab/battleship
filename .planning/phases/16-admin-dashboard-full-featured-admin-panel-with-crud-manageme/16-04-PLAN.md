# Plan 04: Match/content/moderation API

## Objective
Implement match management (list, detail, void, live), content management (emoji CRUD, power-up settings, announcements), and moderation (reports, chat logs, suspicious activity detection) endpoints.

## must_haves
- [ ] Match list with filters, detail view, and void action with point reversal
- [ ] Live match listing via in-memory room state
- [ ] Emoji CRUD and power-up cost/toggle management
- [ ] Announcement CRUD with scheduling
- [ ] Report queue with status transitions (pending → reviewed → resolved/dismissed)
- [ ] Chat log search and suspicious activity detection queries

## Tasks

<action>
In admin-api.js, add match management routes:

GET "/matches" with requireRole("moderator"):
1. Accept query params: page, limit (max 100), dateFrom, dateTo, mode (classic/ranked/all), status (completed/voided/all), playerId (optional user ID filter), sort (default "ended_at"), order (default "desc")
2. Query matches table with LEFT JOIN users for winner/loser display_names
3. Filter by: ended_at BETWEEN dateFrom AND dateTo, mode=$N, voided_at IS NULL/IS NOT NULL, (winner_id=$N OR loser_id=$N)
4. SELECT: m.id, m.winner_id, m.loser_id, w.display_name as winner_name, l.display_name as loser_name, m.mode, m.reason, m.stake, m.started_at, m.ended_at, m.voided_at
5. Return paginated response: { matches: [...], total, page, limit, totalPages }

GET "/matches/:id" with requireRole("moderator"):
1. Query single match with full details: both player names, mode, reason, stake, started_at, ended_at, voided_at, voided_by, void_reason
2. If no row, return 404
3. Return match detail object

POST "/matches/:id/void" with requireRole("admin"):
1. Accept body: reason (required string)
2. Check match exists and is not already voided (voided_at IS NULL)
3. If already voided, return 400 { error: "ALREADY_VOIDED" }
4. UPDATE matches SET voided_at=now(), voided_by=$1, void_reason=$2 WHERE id=$3
5. Reverse point transactions: if match had stake > 0, query transactions WHERE reference_id LIKE 'match_%' AND relates to this match. INSERT reversal transactions (admin_void type) to restore loser's deducted stake and remove winner's payout.
6. Call logAdminAction with action "match.void", target "match", details { reason, matchId, stake }
7. Return 200 { ok: true }

Note: Add voided_at, voided_by, void_reason columns to matches table. If not in 009_admin.sql, add ALTER TABLE matches ADD COLUMN IF NOT EXISTS voided_at TIMESTAMPTZ, voided_by INTEGER REFERENCES users(id), void_reason TEXT at the end of the migration file.

GET "/matches/live" with requireRole("moderator"):
1. Access the rooms object from server.js (passed via mountAdminRoutes(router, io, rooms))
2. Iterate rooms, filter where room.started === true
3. Return array: { roomCode, player1: { clientId, userId, display_name }, player2: { ... }, mode, startedAt, turnCount }
4. Place this route BEFORE "/matches/:id" to avoid :id matching "live"
</action>

<acceptance_criteria>
- GET /api/admin/matches returns paginated match list with player names
- Date range filter correctly bounds results between dateFrom and dateTo
- GET /api/admin/matches/123 returns full match detail
- POST /api/admin/matches/123/void with reason reverses point transactions and marks match voided
- Voiding already-voided match returns 400 ALREADY_VOIDED
- GET /api/admin/matches/live returns currently active rooms from memory
- migration 009_admin.sql includes ALTER TABLE matches for void columns
</acceptance_criteria>

GET "/content/emojis" with requireRole("moderator"):

2. Return { emojis: [...] }

PUT "/content/emojis/:id" with requireRole("admin"):
1. Accept body: name (optional), cost (optional integer >= 0), animation_url (optional), active (optional boolean)
2. Build dynamic UPDATE on premium_emojis table for provided fields only
3. If no row affected, return 404
4. Call logAdminAction with action "content.emoji_edit"
5. Return 200 { ok: true, emoji: updated_row }

POST "/content/emojis" with requireRole("admin"):
1. Accept body: name (required), cost (required integer >= 0), animation_url (required), active (default true)
2. INSERT INTO premium_emojis (name, cost, animation_url, active) VALUES ($1,$2,$3,$4) RETURNING *
3. Call logAdminAction with action "content.emoji_create"
4. Return 201 { ok: true, emoji: new_row }

DELETE "/content/emojis/:id" with requireRole("admin"):
1. DELETE FROM premium_emojis WHERE id=$1
2. If no row affected, return 404
3. Call logAdminAction with action "content.emoji_delete"
4. Return 200 { ok: true }

GET "/content/powerups" with requireRole("moderator"):
1. Query runtime_config WHERE key LIKE 'powerup_%' to get power-up settings
2. If no rows, return default power-up config as JSON (sonar_ping, cross_missile, decoy, scatter_blast with default costs and enabled status)
3. Return { powerups: [...] }

PUT "/content/powerups/:key" with requireRole("admin"):
1. Accept body: cost (optional integer), enabled (optional boolean)
2. UPSERT into runtime_config: INSERT INTO runtime_config (key, value, updated_by) VALUES ('powerup_' || $1, $2::jsonb, $3) ON CONFLICT (key) DO UPDATE SET value=$2::jsonb, updated_by=$3, updated_at=now()
3. Call logAdminAction with action "content.powerup_edit"
4. Return 200 { ok: true }
</action>

<action>
Add moderation routes:

GET "/moderation/reports" with requireRole("moderator"):
1. Accept query: status (pending/reviewed/resolved/dismissed/all, default "pending"), page, limit
2. Query reports JOIN users (reporter + reported display_names), ORDER BY created_at DESC
3. Return { reports: [...], total, page, limit, totalPages }

GET "/moderation/reports/:id" with requireRole("moderator"):
1. Query single report with reporter name, reported name, match_id if linked
2. Return full report detail

PUT "/moderation/reports/:id" with requireRole("moderator"):
1. Accept body: status (required: "reviewed", "resolved", "dismissed"), resolution (optional text — admin notes)
2. Validate status transition: pending → reviewed/resolved/dismissed, reviewed → resolved/dismissed
3. UPDATE reports SET status=$1, reviewed_by=$2, reviewed_at=now(), resolution=$3 WHERE id=$4
4. Call logAdminAction with action "moderation.report_" + status (e.g., "moderation.report_resolve")
5. Return 200 { ok: true }

GET "/moderation/chat" with requireRole("moderator"):
1. Accept query: roomCode (optional), userId (optional), flagged (optional boolean), dateFrom, dateTo, page, limit
2. Query chat_logs with filters, JOIN users for sender display_name
3. Return { messages: [...], total, page, limit, totalPages }

POST "/moderation/chat/:id/flag" with requireRole("moderator"):
1. UPDATE chat_logs SET flagged=true WHERE id=$1
2. Return 200 { ok: true }

GET "/moderation/suspicious" with requireRole("moderator"):
1. Run suspicious activity detection query: users with win_rate > 90% over 20+ games
2. Query: SELECT user_id, COUNT(*) FILTER (WHERE is_winner) as wins, COUNT(*) as total FROM (SELECT winner_id as user_id, true as is_winner FROM matches UNION ALL SELECT loser_id, false FROM matches) sub GROUP BY user_id HAVING COUNT(*) >= 20 AND (COUNT(*) FILTER (WHERE is_winner))::float / COUNT(*) > 0.9
3. JOIN users for display_name
4. Return { suspicious: [{ userId, displayName, wins, total, winRate }] }
</action>

<acceptance_criteria>
- GET /api/admin/moderation/reports?status=pending returns pending reports with player names
- PUT /api/admin/moderation/reports/1 with { status: "resolved", resolution: "banned user" } updates report
- Invalid status transition returns 400
- GET /api/admin/moderation/chat returns searchable chat history
- POST /api/admin/moderation/chat/1/flag marks message as flagged
- GET /api/admin/moderation/suspicious returns users with >90% win rate over 20+ games
- All moderation actions are audit-logged
</acceptance_criteria>
