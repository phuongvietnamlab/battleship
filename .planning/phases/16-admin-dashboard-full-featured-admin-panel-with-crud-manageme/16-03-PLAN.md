# Plan 03: User management API

## Objective
Implement all user management CRUD endpoints under /api/admin/users — list, detail, edit, ban/unban, mute/unmute, point adjustment, delete, and CSV export.

## must_haves
- [ ] Paginated user list with search and sort
- [ ] User detail view with full profile, auth methods, match history, wallet, ban history
- [ ] Ban/unban and mute/unmute with reason logging and WebSocket force-disconnect
- [ ] Point adjustment with mandatory reason and audit trail
- [ ] Soft-delete with PII anonymization
- [ ] CSV export of filtered user list

## Tasks

### Task 1: User list endpoint (GET /api/admin/users)

<read_first>
- admin-api.js
- db.js
</read_first>

<action>
In admin-api.js, inside mountAdminRoutes(router), add GET "/users" handler with requireRole("moderator"):

1. Extract query params: page (default 1), limit (default 25, max 100), search (optional string), sort (default "created_at"), order (default "desc"), status (optional: "active", "banned", "deleted")
2. Build a base query against users table with LEFT JOIN on user_bans (active=true) and LEFT JOIN on admin_roles
3. If search is provided, add WHERE clause: users.display_name ILIKE '%' || $N || '%' OR users.email ILIKE '%' || $N || '%' OR users.id::text = $N
4. If status is "banned", add WHERE condition: EXISTS active ban. If "deleted", WHERE deleted_at IS NOT NULL. If "active", WHERE deleted_at IS NULL AND no active ban.
5. Allowed sort columns whitelist: ["id", "display_name", "email", "created_at", "guest_migrated_at"]. Reject unknown sort values by defaulting to "created_at". Order must be "asc" or "desc" only.
6. Execute COUNT(*) query for total, then SELECT with LIMIT/OFFSET
7. SELECT columns: u.id, u.display_name, u.email, u.avatar_url, u.created_at, u.guest_migrated_at, u.deleted_at, active ban type (if any), admin role (if any)
8. Return JSON: { users: [...], total, page, limit, totalPages: Math.ceil(total/limit) }
9. All search terms use parameterized queries with $N binding
</action>

<acceptance_criteria>
- GET /api/admin/users returns paginated list with total count
- Search by name, email, or ID works (ILIKE for strings, exact for ID)
- Sort by allowed columns only; unknown sort values default to created_at
- Status filter correctly filters active/banned/deleted users
- Response includes totalPages for frontend pagination
- SQL uses parameterized queries — no string concatenation of search term
</acceptance_criteria>

### Task 2: User detail endpoint (GET /api/admin/users/:id)

<read_first>
- admin-api.js
- db.js
</read_first>

<action>
Add GET "/users/:id" handler with requireRole("moderator"):

1. Parse req.params.id as integer. If NaN, return 400 { error: "INVALID_ID" }
2. Query users table: SELECT id, display_name, email, avatar_url, created_at, guest_migrated_at, deleted_at, email_verified FROM users WHERE id=$1
3. If no row, return 404 { error: "USER_NOT_FOUND" }
4. Query credentials: SELECT type, external_id, created_at FROM credentials WHERE user_id=$1 (return as authMethods array, omit password_hash)
5. Query wallets: SELECT balance FROM wallets WHERE user_id=$1 (return as wallet: { balance } or null)
6. Query transactions: SELECT type, amount, balance_after, reference_id, created_at FROM transactions WHERE user_id=$1 ORDER BY created_at DESC LIMIT 50 (return as recentTransactions)
7. Query matches: SELECT COUNT(*) as total, COUNT(*) FILTER (WHERE winner_id=$1) as wins FROM matches WHERE winner_id=$1 OR loser_id=$1 (return as matchStats: { total, wins, losses })
8. Query user_bans: SELECT id, type, reason, starts_at, ends_at, active, banned_by, created_at FROM user_bans WHERE user_id=$1 ORDER BY created_at DESC (return as banHistory)
9. Query admin_roles: SELECT role, granted_at FROM admin_roles WHERE user_id=$1 AND revoked_at IS NULL (return as adminRole or null)
10. Return combined JSON object with all above data
</action>

<acceptance_criteria>
- GET /api/admin/users/123 returns full user profile with auth methods, wallet, match stats, ban history
- Password hashes are never included in the response
- Non-numeric ID returns 400
- Non-existent user returns 404
- matchStats correctly computes wins and losses from matches table
- banHistory includes both active and historical bans
</acceptance_criteria>

- db.js

<action>
Add PUT "/users/:id" handler with requireRole("admin"):

1. Parse req.params.id as integer
2. Accept body fields: display_name (optional string, max 40 chars), avatar_url (optional string), email (optional string)
3. Validate: if display_name provided, sanitize with same logic as db.js sanitizeDisplayName (strip control chars, collapse whitespace, trim, cap 40, HTML-escape)
4. If email provided, normalize (trim+lowercase), validate format with regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/
5. Build dynamic UPDATE query for users table — only SET provided fields. Use parameterized array building.
6. Execute UPDATE users SET ... WHERE id=$N RETURNING id

8. Call logAdminAction(req.adminUser.id, "user.edit", req, "user", String(id), { fields: Object.keys(body that were updated) })
9. Return 200 { ok: true, user: { id, display_name, avatar_url, email } }
</action>

<acceptance_criteria>

- Edit action is logged to admin_audit_log with field names in details
- Non-existent user returns 404
- Display name is sanitized (HTML-escaped, control chars stripped)

### Task 4: Ban/unban and mute/unmute endpoints

<read_first>
- admin-api.js
- admin-auth.js
- server.js
</read_first>

<action>
Add four endpoints, all with requireRole("moderator"):

POST "/users/:id/ban":
1. Accept body: reason (required string), duration (optional string like "7 days", "30 days", "permanent")
2. Validate reason is non-empty string
3. Compute ends_at: if duration is "permanent" or not provided, ends_at = NULL. Otherwise parse as PostgreSQL interval string and compute starts_at + interval.
4. INSERT INTO user_bans (user_id, type, reason, duration, starts_at, ends_at, active, banned_by) VALUES ($1, 'ban', $2, $3::interval, now(), CASE WHEN $3 IS NULL THEN NULL ELSE now() + $3::interval END, true, $4)
5. Force-disconnect: iterate io.of("/").sockets, find sockets where socket.data.userId === userId or socket.data.clientId matches the user's guest credential. Call socket.emit("banned", { reason }) then socket.disconnect(true).
6. Call logAdminAction with action "user.ban", target_type "user", details { reason, duration }
7. Return 200 { ok: true }

POST "/users/:id/unban":
1. UPDATE user_bans SET active=false, unbanned_by=$1, unbanned_at=now() WHERE user_id=$2 AND type='ban' AND active=true
2. Call logAdminAction with action "user.unban"
3. Return 200 { ok: true }

POST "/users/:id/mute":
1. Same pattern as ban but type='mute'. Accept reason and duration.
2. INSERT INTO user_bans with type='mute'
3. Call logAdminAction with action "user.mute"
4. Return 200 { ok: true }

POST "/users/:id/unmute":
1. UPDATE user_bans SET active=false, unbanned_by=$1, unbanned_at=now() WHERE user_id=$2 AND type='mute' AND active=true
2. Call logAdminAction with action "user.unmute"
3. Return 200 { ok: true }

For force-disconnect, the io instance must be accessible. Pass it via mountAdminRoutes(router, io) — update server.js to pass io when calling mountAdminRoutes.
</action>

<acceptance_criteria>
- POST /api/admin/users/5/ban with { reason: "cheating", duration: "7 days" } creates a user_bans row with type='ban', ends_at = now + 7 days
- POST /api/admin/users/5/ban with { reason: "repeat offender" } (no duration) creates permanent ban (ends_at NULL)
- Ban without reason returns 400 { error: "REASON_REQUIRED" }
- POST /api/admin/users/5/unban sets active=false on all active bans for that user
- Mute/unmute follow same pattern with type='mute'
- Force-disconnect emits "banned" event to the user's socket before disconnecting
- All actions logged to admin_audit_log
- Moderators, admins, and super_admins can all execute ban/unban
</acceptance_criteria>

### Task 5: Point adjustment endpoint

<read_first>
- admin-api.js
- admin-auth.js
- db.js
</read_first>

<action>
Add POST "/users/:id/points" handler with requireRole("admin"):

1. Accept body: amount (required integer, can be positive or negative), reason (required string)
2. Validate: amount must be integer and non-zero. Reason must be non-empty string.
3. If amount > 0, call creditWallet(userId, amount, "admin_credit", "admin_adjust_" + req.adminUser.id + "_" + Date.now())
4. If amount < 0, call debitWallet(userId, Math.abs(amount), "admin_debit", "admin_adjust_" + req.adminUser.id + "_" + Date.now())
5. If debit result is { ok: false, code: "INSUFFICIENT_BALANCE" }, return 400 { error: "INSUFFICIENT_BALANCE" }
6. Call logAdminAction with action "user.point_adjust", details { amount, reason, newBalance }
7. Return 200 { ok: true, balance: newBalance }
</action>

<acceptance_criteria>
- POST /api/admin/users/5/points with { amount: 100, reason: "compensation" } credits 100 points
- POST /api/admin/users/5/points with { amount: -50, reason: "refund error" } debits 50 points
- Debit that would go negative returns 400 INSUFFICIENT_BALANCE
- Missing reason returns 400 { error: "REASON_REQUIRED" }
- amount=0 returns 400 { error: "INVALID_AMOUNT" }
- Only admin/super_admin can adjust points (moderator gets 403)
- Action logged with amount and reason in details
</acceptance_criteria>

### Task 6: Delete user and CSV export endpoints

<read_first>
- admin-api.js
- admin-auth.js
- db.js
</read_first>

<action>
Add two endpoints:

DELETE "/users/:id" with requireRole("admin"):
1. Accept query param: hard=true (optional, requires super_admin)
2. If hard=true and req.adminUser.role !== "super_admin", return 403
3. Soft delete (default): UPDATE users SET deleted_at=now(), display_name='[deleted]', email=NULL, avatar_url=NULL WHERE id=$1 AND deleted_at IS NULL
4. If no row affected, return 404 { error: "USER_NOT_FOUND" }
5. Hard delete (if hard=true): DELETE FROM credentials WHERE user_id=$1, DELETE FROM user_bans WHERE user_id=$1, then DELETE FROM users WHERE id=$1
6. Force-disconnect the user (same as ban)
7. Call logAdminAction with action "user.delete", details { hard: Boolean(hard) }
8. Return 200 { ok: true }

GET "/users/export" with requireRole("admin"):
1. Accept same query params as list (search, status) but no pagination
2. Query all matching users (up to 10,000 row cap)
3. Build CSV string with headers: id, display_name, email, created_at, status (active/banned/deleted)
4. Set response headers: Content-Type: text/csv, Content-Disposition: attachment; filename="users-export-YYYY-MM-DD.csv"
5. Call logAdminAction with action "user.export", details { count, filters }
6. Return CSV string as response body

Important: Place the "/users/export" route BEFORE "/users/:id" to avoid :id matching "export".
</action>

<acceptance_criteria>
- DELETE /api/admin/users/5 soft-deletes (sets deleted_at, anonymizes PII)
- DELETE /api/admin/users/5?hard=true with super_admin actually removes rows
- DELETE with hard=true by admin role returns 403
- Soft-deleted user has display_name='[deleted]', email=NULL
- GET /api/admin/users/export returns CSV with Content-Type text/csv
- CSV includes headers and rows for matching users
- Export is capped at 10,000 rows
- Both actions logged to audit log
</acceptance_criteria>
