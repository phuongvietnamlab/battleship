# Phase 5 — UI Design Contract: Public Matchmaking

> Visual and interaction contract for the matchmaking queue UI.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

> 1. Two queue-entry buttons inserted into the existing Lobby screen
> 2. A new `"queue"` screen (wait panel) shown while searching for an opponent

### 3. Match Found Transition

When `matchFound` fires, the screen transitions to `"placement"` immediately (D-10, instant drop-in). No intermediate screen or accept step.

**Transition feedback:** A brief visual signal that a match was found before the placement screen loads:
- Show `.status-pill.pill-ready` with `t("queue.matched")` copy for ~300ms while state settles
- This is handled by setting a transitional state: `queueStatus: "matched"` renders the pill, then `setScreen("placement")` is called in the same tick via `socket.on("matchFound")`
- In practice the pill appears for 0-1 render frames — the key feedback is the immediate switch to the placement screen

Implementation note: The existing `opponentJoined` event already logs `t("log.oppJoined")` — this appears in the placement screen's log, providing additional confirmation that an opponent was found.

## Interaction Contracts

### Queue Entry Flow

1. User taps "Quick Match" or "Ranked" in lobby
2. Client emits `joinQueue({type, clientId, profile})` via socket
3. On `{ok: true}` callback: `setScreen("queue")`, `setQueueSince(Date.now())`, `setQueueType(type)`
4. On error callback: display `<div className="error">{errText(res)}</div>` inline in lobby (no screen change)

### Timer Display

- Computed as `Math.floor((Date.now() - queueSince) / 1000)` inside a `useEffect` with `setInterval(1000)`
- Format: `MM:SS` (e.g., `00:45`, `01:30`) using padded string formatting
- `useRef` interval — cleared on Leave Queue, matched, or component unmount

### Bot Offer Delay

- `botOfferTimerRef` = `useRef(null)` set at enqueue time to `setTimeout(showBotOffer, 30000)`
- 30 seconds (A3 from RESEARCH.md — tunable)
- Cleared on: cancel, matchFound, component unmount
- Applies to BOTH casual and ranked queues (a waiting ranked player is equally lonely)

### Leave Queue

- Tapping Leave Queue emits `leaveQueue()` then resets queue state
- State reset: `setQueueType(null)`, `setQueueSince(null)`, `setQueueWindow(null)`, `setBotOfferVisible(false)`, `setScreen("lobby")`
- Leave Queue has NO confirmation prompt — immediate, non-destructive action

### Navigate Away / Tab Close

- `useEffect` cleanup in queue screen calls `socket.emit("leaveQueue")` (D-12)
- `beforeunload` listener not required — socket disconnect (trigger 1 in RESEARCH.md) handles tab close

### ELO Window Display (Ranked Only)

- Server sends `queueStatus` event: `{waitSec, windowWidth, queueSize}` periodically
- Client updates `queueWindow` state on receipt
- When `windowWidth >= INFINITY` (after cap), display "any rating" instead of `±{N}`
- Initial display: show the starting window (±150 for established, ±300 for provisional) even before the first server update

### Match Found

```
socket.on("matchFound", ({code: matchCode, ranked: isRanked}) => {
  clearBotOfferTimer();
  setCode(matchCode);
  persistRoom(matchCode);
  setQueueType(null);
  setQueueSince(null);
  setQueueWindow(null);
  setBotOfferVisible(false);
  setScreen("placement");  // D-10: instant drop-in
});
```

No loading state, no accept step. The placement screen handles the "waiting for opponent" state via existing `pill-wait` / `pill-ready` status pills.

### Copywriting Principles (from existing app patterns)

- Action labels use verb-first imperative: "Quick Match", "Leave Queue", not "Start Quick Match" or "Exit Queue"
- Status text is present-progressive (what's happening now): "Searching…" not "Search"
- Error text states the problem without blaming the user: "You're already in a queue" not "Invalid request"
- Bot offer is a gentle offer, not a failure message — no "No opponents found" framing
