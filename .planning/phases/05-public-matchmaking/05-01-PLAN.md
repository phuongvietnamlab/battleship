<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server (socket joinQueue/leaveQueue) | Untrusted arg payload (type, clientId, profile) crosses into server queue state |
| queue entry → auto-created room | Queue-controlled data seats players into a real game room |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 | Tampering | joinQueue profile field | mitigate | `sanitizeProfile(arg.profile)` applied at enqueue (same as createRoom line 1278) before storing in entry |
| T-5-02 | Spoofing | joinQueue type field | mitigate | Server normalizes `type` to allowlist `"ranked" | "casual"` only — any other value coerces to casual |
| T-5-03 | Denial of Service | joinQueue spam (join/leave cycling) | mitigate | `joinQueueLimiter` RateLimiterMemory 5/60s per clientId → RATE_LIMITED (existing dep, no new package) |
| T-5-04 | Denial of Service | double-enqueue (client sends joinQueue twice) | mitigate | `ALREADY_IN_QUEUE` guard on socket.data.queueType; `ALREADY_IN_ROOM` guard on socket.data.code |
| T-5-05 | Integrity | double-pairing race (same player matched into two rooms) | mitigate | Synchronous `queues[type].delete()` of both entries before any await in tryPair + `entry.pairing` flag (mirrors room.resolving) |
| T-5-SC | Tampering | npm/pip/cargo installs | accept | No packages added this phase (RESEARCH Package Legitimacy Audit: none); no install task → no legitimacy gate needed |
</threat_model>
